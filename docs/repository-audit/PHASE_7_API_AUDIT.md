# Phase 7: API Audit Report

## 1. Executive Summary
An API audit of the REST and WebSocket endpoints in the Esparex backend was conducted. The endpoints are versioned under the `/api/v1` namespace. The audit discovered high-severity violations of the API Contract SSOT, where several administration/moderation endpoints are registered twice (supporting both `POST` and `PATCH` methods for the same path), and admin-level namespaces are split across multiple mounting configurations.

---

## 2. Scope
This audit evaluated:
- REST endpoint declarations inside `backend/user/src/routes/`
- Namespace layout and versioning
- Alignment of HTTP verbs with method strictness (SSOT Section 2)
- Mount points inside `backend/user/src/app.ts`
- WebSocket connection hooks

---

## 3. Inventory

### REST Endpoint Namespaces (in `app.ts`)
- `/api/v1/auth/*` — User OTP authentication
- `/api/v1/users/*` — User Profile management
- `/api/v1/listings/*` — Classified ads listing engine (vehicles/parts)
- `/api/v1/catalog/*` & `/api/v1/catalog-requests/*` — Category taxonomies
- `/api/v1/locations/*` — Geocoding & autocomplete search
- `/api/v1/payments/*` & `/api/v1/invoices/*` — Checkout orders
- `/api/v1/ai/*` — Generative content descriptions
- `/api/v1/notifications/*` & `/api/v1/smart-alerts/*` — Push/SMS/Email notifications
- `/api/v1/chat/*` — Socket-complemented chat messaging
- `/api/v1/admin/*` — Back-office portal access routes (split into catalog and request segments)

### WebSocket Namespaces
- Socket.io is initialized at server level and handles live messaging connections under default path `/socket.io`.

## 4. Findings

### High Severity Findings
1. **HTTP Method Duplication for Ad Moderation Handlers**
   - **Finding**: Moderation actions on listings (approvals, rejections, extensions, deactivations, expires, resolved reports) are registered twice using both `POST` and `PATCH` verbs:
     - `/listings/:id/approve` (`POST` & `PATCH`)
     - `/listings/:id/reject` (`POST` & `PATCH`)
     - `/listings/:id/deactivate` (`POST` & `PATCH`)
     - `/listings/:id/expire` (`POST` & `PATCH`)
     - `/listings/:id/extend` (`POST` & `PATCH`)
     - `/listings/:id/report-resolve` (`POST` & `PATCH`)
   - **Impact**: Violates Section 2 (HTTP Method Strictness) of the [API Contract SSOT](file:///c:/Users/Administrator/Documents/GitHub/Esparex/docs/ssot/API_CONTRACT_SSOT.md), which specifies that partial status updates must use `PATCH` and creation must use `POST`. This bloats the routing tables and makes API consumption ambiguous.

2. **Split Admin API Mount Surfaces**
   - **Finding**: Instead of mounting all admin routes under `/api/v1/admin/`, the application splits them across three mounting points:
     - `app.use('/api/v1/admin/catalog-requests', adminCatalogRequestRoutes)` (app.ts line 420)
     - `app.use('/api/v1/admin/catalog', adminCatalogRoutes)` (app.ts line 421)
     - `app.use('/api/v1/admin', adminRoutes)` (app.ts line 422)
   - **Impact**: Violates Section 1.2 of the [API Contract SSOT](file:///c:/Users/Administrator/Documents/GitHub/Esparex/docs/ssot/API_CONTRACT_SSOT.md), which mandates that all administrative endpoints reside under `/api/v1/admin/*`. This split complicates path gating and authentication middleware enforcement.

---

### Medium Severity Findings
3. **Implicit OpenAPI Swagger Divergence**
   - **Finding**: The server generates OpenAPI JSON schemas at `/api-docs.json` for developer reference (only in dev/staging), but there is no runtime validation layer matching incoming requests/responses to these schemas.
   - **Impact**: Minor mismatch drift between types generated by frontend Zodios clients and the actual schemas sent by Express controllers.

---

## 6. Risk Level
- **Overall API Risk**: **Medium**
- The endpoints are securely versioned under `/api/v1`, but method verb replication and route splits increase API inconsistency.

---

## 7. Recommendations
1. **Prune Duplicated Verbs**: Choose one canonical verb for moderation actions (preferably `PATCH` as they represent partial updates of the listing status) and delete the duplicate `POST` routes from `adminRoutes.ts`.
2. **Consolidate Admin Routes**: Mount `adminCatalogRequestRoutes` and `adminCatalogRoutes` inside `adminRoutes.ts` directly, then mount `adminRoutes` at `/api/v1/admin` in `app.ts`. This routes all requests through a single entry gate.
